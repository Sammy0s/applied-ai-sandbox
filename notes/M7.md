## Plan from chat
Notes Pinning Feature — Design PlanNotes Pinning Feature — Design Plan
Data Model
Add one field to the existing Note record:

is_pinned — boolean, default false

That's it. No new tables, no separate pinned-notes collection.

Routes / API
Two changes to your existing notes endpoint:

PATCH /notes/:id — if it doesn't already exist, add support for updating is_pinned: true/false. If you already have a general update route, this might require zero new code.
GET /notes (home page) — sort results so pinned notes come first, then the rest by their usual order (created date, etc.). Handle this in the query, not in the frontend.


UI
Home page

Pinned notes render in a visually distinct section at the top, labeled "Pinned" (a simple header is enough).
Unpinned notes follow in their normal list below.
If there are no pinned notes, the "Pinned" section doesn't appear at all.

Pin/unpin control

A pin icon on each note card (📌 or similar). Filled = pinned, outline = unpinned.
Clicking it toggles the state immediately (optimistic update), calls the PATCH route.
No confirmation dialog needed — unpinning is just one click away.

What's deliberately out of scope

Drag-to-reorder pinned notes
Pinning to specific positions
Per-user pin state
Any kind of "pin limit" UI
