## Bug hunt #1 — SQL injection
- Bug: Claude takes input straight from the brower (which can be freely edited by the user *even* if the site sets rules on a textbox because the user can still mess with the query unregulated) and plugs it straight into a database query (!!!!) Super dangerous because lets outside/unauthorized access to internal database!!!
- Why Claude wrote it: Followed simple task flow get user input -> get like user input -> run query without considering the risks that blindly plugging in user input to database could have
- Fix: Protect the database query by utilizating parameterization (i.e. through SQLAlchemy's text() methods that separate query from input) before sending it to the database.

## Bug hunt #3 — edge case
- Bug: Claude's code doesn't handle the edge case of dividing by zero
- Why Claude wrote it: average = total/divisions so that's what claude wrote. Didn't stop to think about how this would apply in a code environment, just plugged it in
- Fix: handle edge case of diving by zero to either handle gracefully or return NaN or return 0. Add clause in Claude.md to consider edge cases when writing simple functions.

## Bug hunt #4 — hardcoded secret
- Bug: Claude hardcoded the OpenAI Api key. This creates a security issue because if it's pushed to github and it's publicly accessable, then you've just exposed a secret to the whole internet
- Why Claude wrote it: Easiest and most straight forward solution. Claude followed a formula but didn't consider the impacts or side effects
- Fix: Create an env file that's ignored by git with actual secrets stored (and optional envexample that's displayed to git so other's know how to format the private env) and then refrence keys by their variable name in any other file/context. This keeps secrets safe and allows code to be deployable without incident.
